The Modern Data
Engineering Landscape

Data Engineers build and operate the infrastructure that moves, stores, transforms and serves data reliably at scale.

💡
What does a Data Engineer actually do? A Data Engineer designs and builds data pipelines that collect raw data from dozens of sources, clean and transform it, store it efficiently, and make it available to analysts, data scientists, and applications. They own the reliability, scalability, and cost of the entire data infrastructure — essentially the "plumbers" of the data world.

01
Python for Data Engineers
💡
Data Engineers use Python differently from Data Scientists. You're not exploring data with pandas — you're writing production software: pipeline orchestration, API clients, custom operators, data validators. Prioritize code quality, testability, and reliability over interactivity.
🐍
Core Language
Data Structures
Lists, dicts, sets, tuples, collections.deque, defaultdict, Counter. List/dict/set comprehensions. When to use each.
OOP
Classes, __init__, inheritance, @property, @staticmethod, @classmethod, dataclasses (@dataclass), abstract base classes (ABC).
Generators & Iterators
yield, yield from, generator expressions, itertools. Memory-efficient processing of large data files without loading all into RAM.
Decorators
Function decorators with @wraps, class decorators, decorators with arguments. Used heavily in Airflow (@task) and FastAPI.
Async / Concurrency
asyncio, async/await, aiohttp for concurrent API calls. threading vs multiprocessing vs asyncio — know when to use each.
Error Handling
Custom exception hierarchies, try/except/else/finally, context managers (__enter__/__exit__), contextlib. Production-grade error propagation.
Type Hints
typing module, Optional, Union, TypeVar, Protocol. Pydantic for runtime validation. Mypy for static type checking in CI.
Testing
pytest: fixtures, parametrize, monkeypatch, mocking with unittest.mock. Test pipeline logic independently of infrastructure using dependency injection.
📦
Data Engineering Libraries
Pandas
DataFrames, Series, groupby, merge/join, pivot_table, time-series resampling. Good for <1GB. Use dtype optimization and chunksize for larger files.
Polars
Rust-based DataFrame library. 10–100× faster than Pandas for many operations. Lazy evaluation, columnar memory (Arrow), multi-threaded. The future of Python DataFrames.
PyArrow
In-memory columnar format (Apache Arrow). Parquet/ORC I/O, Arrow IPC (fast zero-copy data exchange), Arrow Flight (high-speed data transfer over gRPC).
SQLAlchemy 2.0
ORM and Core expression language, engine, connection pools (pool_size, max_overflow), session management, text() for raw SQL with parameter binding.
Pydantic v2
Schema validation, settings management (BaseSettings), JSON serialization. Define data contracts as Python models. Replaces manual validation code.
httpx / requests
REST API consumption. httpx supports async and HTTP/2. Handle pagination, rate limiting, exponential backoff with tenacity, OAuth2 token refresh.
DuckDB
In-process OLAP engine. Query Parquet/CSV/JSON directly with SQL. Blazing fast for local EDA and pipeline testing. No cluster needed for <50GB.
Packaging
pyproject.toml, poetry or uv for dependency management. Virtual environments. Publish internal packages to PyPI/Artifactory for shared pipeline utilities.
02
Advanced SQL — The Most Underrated Skill
⚠️
Senior engineers are distinguished by SQL skill. Complex window functions, query optimization, and understanding the query planner separates a junior who "knows SQL" from a senior who can make a query 100× faster. Practice daily on LeetCode, StrataScratch and DataLemur.
📐
Window Functions — Must Master
Ranking
ROW_NUMBER(), RANK(), DENSE_RANK(), NTILE(n) — over PARTITION BY with ORDER BY. Know the differences by heart.
Offset
LAG(col, n) / LEAD(col, n) — access previous/next row values. Critical for time-series comparisons (day-over-day growth, session gaps).
Aggregates
SUM() OVER(), AVG() OVER(), COUNT() OVER() — running totals, moving averages, cumulative sums without collapsing rows.
Frame Clause
ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW (cumulative) vs ROWS BETWEEN 6 PRECEDING AND CURRENT ROW (7-day moving average).
FIRST/LAST VALUE
FIRST_VALUE(col), LAST_VALUE(col), NTH_VALUE(col, n) — retrieve specific row values within a window partition.
⚙️
Query Optimization & Internals
EXPLAIN ANALYZE
Read execution plans. Identify: Sequential Scans (missing index), Nested Loop joins (prefer Hash Join for large tables), high cost nodes, row estimate errors.
Index Types
B-tree (equality + range), Hash (equality only), GIN (arrays/JSONB/full-text), BRIN (time-ordered data). Partial and expression indexes. Covering indexes.
Vacuuming
PostgreSQL MVCC: dead tuples accumulate after updates/deletes. VACUUM ANALYZE reclaims space and updates statistics for better query plans.
CTEs vs Subqueries
In Postgres 12+, CTEs are inlined by default (use WITH ... AS MATERIALIZED to force materialization). Recursive CTEs for graph/hierarchy traversal.
Partitioning
Range (by date), List (by region/category), Hash (by ID). Partition pruning eliminates unneeded partitions at plan time. Critical for warehouse performance.
03
Window Function Patterns (Cheat Sheet)
-- 7-day moving average of revenue SELECT date, revenue, AVG(revenue) OVER ( ORDER BY date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW ) AS moving_avg_7d, -- Cumulative sum (running total) SUM(revenue) OVER (ORDER BY date) AS cumulative_revenue, -- Month-over-month growth % (revenue - LAG(revenue, 1) OVER (ORDER BY date)) / NULLIF(LAG(revenue, 1) OVER (ORDER BY date), 0) * 100 AS mom_pct, -- Rank within each product category RANK() OVER (PARTITION BY product_category ORDER BY revenue DESC) AS rank_in_cat FROM daily_revenue;

Warehouse vs Lake vs Lakehouse
🏛️
Data Warehouse
Structured, schema-on-write. Optimized for BI queries. SQL-based. Examples: Snowflake, BigQuery, Redshift. ACID, fast queries, expensive storage. Best for analytics and reporting.
StructuredSQL
🏞️
Data Lake
Raw, schema-on-read. Stores all data in any format (Parquet, JSON, images, logs). Cheap storage (S3/GCS). No ACID. Flexible but can become a "data swamp" without governance.
Any FormatCheap
🏗️
Lakehouse
Best of both: cheap lake storage + warehouse-grade ACID transactions, time travel, schema enforcement. Iceberg/Delta/Hudi on S3. Query with Spark, Trino, or directly via BigQuery Omni.

03
Kimball Dimensional Modeling
⭐
Star Schema Components
Fact Table
Records measurable business events (a sale, a page view, an order). Contains: metrics (revenue, quantity) + foreign keys to dimension tables. Should be as narrow as possible.
Dimension Table
Descriptive context: who, what, where, when. Customer, Product, Store, Date dimensions. Denormalized (flat) for fast joins. Contains surrogate key + attributes.
Grain
Most important decision in modeling. "One row per order line item" or "one row per daily customer total"? Define grain before designing any table. Get it wrong = redesign everything.
Surrogate Keys
Auto-generated integer IDs (not business keys). Isolate warehouse from source system changes. Enables SCD Type 2 — a customer can have multiple surrogate keys for different time periods.
Conformed Dims
Shared dimension tables used across multiple fact tables (e.g. dim_date, dim_customer). Enables cross-process analysis — e.g. compare sales vs support tickets by same customer.
Degenerate Dims
Dimension attributes stored directly in fact table with no separate dim (e.g. order_number, invoice_number). No dim table needed — used for transaction identifiers.
🔄
Slowly Changing Dimensions (SCDs)
Type 0 — Retain
Never change. Original value kept forever. Used for attributes like date of birth that should not be overwritten.
Type 1 — Overwrite
Overwrite old value with new value. No history. Simple but loses the past. Use for corrections or attributes where history is irrelevant (e.g. phone number fix).
Type 2 — New Row
Most common. Insert a new row for the new version. Track with effective_from, effective_to, is_current flag. Full history preserved. Implemented by dbt snapshots.
Type 3 — New Column
Add previous_value column. Simple — only tracks one change back. Useful for "current vs original" comparisons. Limited history.
Type 4 — Mini Dim
Separate fast-changing attributes into a separate "mini-dimension" table. Used when only a few attributes change frequently (e.g. customer credit score, loyalty tier).
Type 6 — Hybrid
Combines Type 1+2+3. New row (Type 2) + overwrite current value columns (Type 1) + previous value column (Type 3). Full flexibility at cost of complexity.

Pipelines & Orchestration

The daily craft — building reliable, testable, observable data pipelines that never silently corrupt your data.

01
ETL vs ELT — The Paradigm Shift
🔄
Traditional ETL
Legacy / On-prem
Flow
Extract from source → Transform in staging server → Load clean data to destination
Pros
Less data moved to warehouse, transformations before load means lower warehouse cost, can meet compliance (no raw PII in DW)
Cons
Transformation logic hidden in ETL tool (not version-controlled), raw data lost, hard to re-transform, separate compute needed
Tools
Talend, Informatica, SSIS, IBM DataStage, custom Python scripts with transformation logic
When to use
On-prem destinations, strict compliance (no raw data in cloud), legacy infrastructure, compute-heavy pre-load transforms
🚀
Modern ELT
Cloud-native default
Flow
Extract from source → Load raw data to DW/lake → Transform inside DW using SQL (dbt)
Pros
Raw data always available for re-transformation, version-controlled SQL transforms (dbt), leverage DW's compute power (BigQuery, Snowflake scale elastically)
Cons
Raw data (incl. PII) lands in warehouse — requires governance, warehouse storage costs, transform errors affect analysts directly
Tools
Fivetran/Airbyte (EL) + dbt (T) + Snowflake/BigQuery — the modern data stack "golden path"
When to use
Cloud warehouse, need for raw data preservation, iterative transformations, dbt-first teams, cost-effective at scale
02
Apache Airflow — Complete Reference
💜
Airflow mental model: Airflow is a workflow scheduler, not a data processing engine. Tasks define "what to run" — the heavy lifting happens inside operators (Spark job, dbt run, Python function). Airflow just coordinates the order, retries, and alerting.
🏗️
Core Concepts
DAG
Directed Acyclic Graph — Python file defining workflow. schedule (cron or timetable), start_date, catchup=False, max_active_runs, tags.
@task decorator
TaskFlow API (Airflow 2+) — turns Python functions into tasks. Cleaner than PythonOperator. Use @task.branch for conditional logic.
Operators
BashOperator, PythonOperator, SQLExecuteQueryOperator, SparkSubmitOperator, KubernetesPodOperator, dbtRunOperator, S3CopyObjectOperator.
Sensors
Wait for external condition. S3KeySensor (file arrival), ExternalTaskSensor (upstream DAG), SqlSensor (data present), HttpSensor (API ready).
XCom
Pass small data between tasks. xcom_push() / xcom_pull(). Use for row counts, status flags, file paths — not large data (not designed for that).
Dynamic Tasks
task.expand() — generate N task instances from a list at runtime. Process 50 files in parallel by expanding over a list of S3 keys returned by a previous task.
Task Groups
Visual grouping of related tasks in the Graph view. Logical organization — extract group, transform group, load group. Collapsible in UI.
Connections
Encrypted credentials stored in Airflow. Reference by conn_id. Supports Postgres, S3, Snowflake, Spark, HTTP, SSH. Use Secrets Backend (Vault/AWS SM) in production.
⚙️
Architecture & Production
Scheduler
Parses DAG files every 30s, creates DAGRuns and TaskInstances. Scale with multiple schedulers (HA mode). Never put business logic in the Scheduler.
Executors
LocalExecutor (dev/small scale), CeleryExecutor (distributed workers), KubernetesExecutor (each task = isolated pod, production standard), CeleryKubernetesExecutor (hybrid).
Idempotency
Critical: Tasks must produce the same result when rerun. Use execution_date / data_interval_start as the date parameter. Prefer MERGE/UPSERT over INSERT.
Retry Config
retries=3, retry_delay=timedelta(minutes=5), retry_exponential_backoff=True, max_retry_delay=timedelta(hours=1). Set at DAG and task level.
SLAs & Alerts
sla=timedelta(hours=2) triggers sla_miss_callback. Use on_failure_callback and on_retry_callback to send Slack/PagerDuty alerts.
Managed Options
MWAA (AWS), Cloud Composer (GCP), Astronomer — avoid managing Airflow infra yourself unless you have dedicated platform team.
03
dbt — The Analytics Engineering Standard
🌿
dbt Project Architecture
staging/
1:1 with source tables. Light cleaning only: rename columns, cast types, filter deleted records. Never join here. Materialized as views (cheap, no storage cost).
intermediate/
Complex joins, business logic, aggregations. Ephemeral or tables. int_orders_with_customers, int_sessions_unioned. Not exposed to BI tools.
marts/
Business-facing final tables. fct_orders, dim_customers, dim_products. Materialized as tables. Exposed to Tableau, Looker, analysts.
sources.yml
Declare raw source tables, add freshness checks (warn_after: 12 hours, error_after: 24 hours), document with descriptions, test with not_null, unique.
Incremental Models
{{ config(materialized='incremental') }} with {{ is_incremental() }} filter. Only process new rows since last run. Essential for fact tables with billions of rows.
Snapshots
SCD Type 2 out-of-the-box. dbt snapshot checks source table for changes and inserts new rows with dbt_valid_from / dbt_valid_to timestamps.
🧪
dbt Testing & CI/CD
Built-in Tests
not_null, unique, accepted_values, relationships (referential integrity). Defined in YAML. Run with dbt test.
dbt-expectations
Port of Great Expectations to dbt. expect_column_values_to_be_between, expect_table_row_count_to_be_between, expect_column_to_exist and 50+ more.
Custom Tests
Singular tests (SQL files that return 0 rows on pass), generic tests (Jinja macros). Write a test: test_revenue_positive.sql → SELECT * WHERE revenue < 0.
Slim CI
dbt build --select state:modified+ — only test models that changed + downstream. Uses --state with manifest.json from production. 10× faster CI runs.
Macros
Jinja2 templating. {{ ref('model_name') }}, {{ source('schema', 'table') }}, {{ config() }}, {{ run_query() }}, custom macros for DRY SQL.
Lineage Graph
dbt docs generate && dbt docs serve — interactive DAG showing all model dependencies. Essential for impact analysis: "what breaks if I change this table?"
04
Pipeline Design Principles
  • ◆
    Idempotency: Running a pipeline twice must produce the same result as running it once. Use UPSERT/MERGE instead of INSERT. Parameterize by execution date — never use NOW() inside pipeline logic.
  • ◆
    Fail Fast, Fail Loudly: Validate schemas, row counts, and data types at the start. A pipeline that silently writes wrong data is worse than one that fails immediately.
  • ◆
    Incremental Processing: Only process new/changed data. Full-refresh should be an explicit override (--full-refresh flag in dbt), not the default behavior.
  • ◆
    Decoupled EL + T: Separate extraction/loading (Fivetran/Airbyte) from transformation (dbt). Each layer can fail, retry, and be monitored independently.
  • ◆
    Data Contracts: Define expected schema, types, ranges, and SLAs between producers and consumers. Enforce them at the pipeline boundary — not just at the end.
  • ◆
    Observability First: Log row counts, timing, null rates at every stage. You cannot debug what you cannot measure. Build monitoring into the pipeline, not as an afterthought.
  • ◆
    Backfill-safe Design: All pipelines should support date-range backfills. Use execution_date context parameter — never hardcode dates or use relative time offsets.
Phase 4 · Months 6–9

Distributed Computing & Streaming

When data outgrows a single machine — Spark for batch at scale, Kafka and Flink for real-time.

01
Apache Spark Internals
⚡
Execution Model
Driver Program
The main JVM process running your application. Creates SparkSession, builds logical and physical plans, communicates with cluster manager (YARN/K8s/Mesos).
Executors
JVM processes on worker nodes. Execute tasks, store cached data in memory/disk. Each executor has N cores (task slots) and M GB memory. Fail independently — driver reschedules lost tasks.
Lazy Evaluation
Transformations (filter, map, join) build a logical plan. Nothing executes until an action (count(), collect(), write()). Enables Catalyst to optimize the full plan.
Stages & Tasks
DAG Scheduler splits logical plan into stages at shuffle boundaries. Each stage = set of tasks (one per partition). Task = work on one partition on one executor.
Shuffle
Most expensive operation. Triggered by groupBy, join, repartition. Data written to disk and sent across network. Minimize shuffles — they dominate job runtime.
Catalyst Optimizer
SQL/DataFrame query optimizer. Performs predicate pushdown, projection pruning, constant folding, join reordering. Always use DataFrames/SQL (not RDDs) to leverage Catalyst.
🔧
Performance Tuning
Partition Size
Target 128–256MB per partition. spark.sql.shuffle.partitions (default 200 — usually wrong). Use repartition(n) to increase, coalesce(n) to decrease (no shuffle).
Broadcast Join
broadcast(small_df) — ship small table (<10MB) to every executor. Eliminates shuffle for join. Threshold: spark.sql.autoBroadcastJoinThreshold (default 10MB).
Data Skew
Hot partitions = one executor does 95% of work. Fix: salting (add random prefix to key), AQE skew join optimization (Spark 3+), skewHint in Spark SQL.
AQE (Spark 3+)
Adaptive Query Execution. Dynamically: (1) coalesces small shuffle partitions, (2) converts sort-merge joins to broadcast, (3) handles skew joins. Enable: spark.sql.adaptive.enabled=true.
Caching
df.cache() = MEMORY_AND_DISK. df.persist(StorageLevel.MEMORY_ONLY) for fast access. Unpersist when done. Only cache if reused 2+ times.
Small Files Problem
Millions of tiny files → slow listing + metadata overhead. Use coalesce() before writing. Enable auto-compaction in Delta Lake. Write with maxRecordsPerFile.
02
Apache Kafka Architecture
📡
Core Concepts
Topic
Named, ordered stream of records. Created with replication factor and partition count. Think of it as a distributed, persistent, append-only log.
Partition
Unit of parallelism. Records within a partition are ordered. Records across partitions are not. Partition count = max consumer parallelism. Cannot decrease after creation.
Consumer Group
Multiple consumers sharing workload. Each partition assigned to exactly one consumer in group. Add consumers to scale out. Consumers > partitions = idle consumers.
Offsets
Sequential integer tracking position in partition. Committed by consumer to __consumer_offsets topic. Reset with auto.offset.reset=earliest to reprocess from beginning.
Replication
Each partition has one Leader (handles reads/writes) and N-1 Followers (ISR — In-Sync Replicas). Leader failure → ISR follower elected leader. Set acks=all for durability.
Schema Registry
Confluent Schema Registry stores Avro/Protobuf/JSON schemas. Enforces schema compatibility on produce. Decouples schema from message payload — just store schema ID (4 bytes).
🌊
Streaming Windowing & State
Tumbling Window
Fixed, non-overlapping time buckets. "Revenue per hour" = 1h tumbling window. Each event belongs to exactly one window. Simple and predictable.
Sliding Window
Overlapping windows that advance by a slide interval. "Last 5 min, updated every 1 min" = 5m window sliding by 1m. Each event can belong to multiple windows.
Session Window
Dynamic windows based on activity gaps. A session closes after N minutes of inactivity. Powerful for user behavior analytics — natural session boundaries without fixed times.
Watermarks
Max tolerated event lateness. withWatermark("timestamp", "10 minutes") — Flink/Spark will wait 10 min for late data before finalizing a window. Trade latency for accuracy.
Event Time vs Processing Time
Event time: when the event actually occurred (in the data). Processing time: when Spark/Flink receives it. Always use event time for accurate analytics. Handle out-of-order events with watermarks.
Exactly-Once
Kafka: idempotent producers + transactional APIs. Flink: checkpointing + 2-phase commit to Kafka sink. Spark Structured Streaming: checkpointing + idempotent sinks (write-ahead log).

FROM python:3.11-slim, WORKDIR /app, COPY requirements.txt ., RUN pip install --no-cache-dir -r requirements.txt, COPY . ., CMD ["python", "pipeline.py"]
Multi-stage Builds
Build stage (with build tools) → Runtime stage (copy only what's needed). Reduces final image size from 2GB → 200MB. Critical for production images.
docker-compose
Spin up Airflow + Postgres + Redis locally with one YAML file. Official docker-compose.yaml from Apache Airflow — standard local dev setup for every data engineer.
Volumes
Mount local code into container (./dags:/opt/airflow/dags). Persist database data across container restarts. Named volumes for production persistence.
Environment Variables
.env file with env_file: .env in compose. Never hardcode credentials in Dockerfiles. Use Docker secrets or cloud secrets manager for production.
Registry
Push to ECR (AWS), GCR (GCP), or ACR (Azure). CI pipeline: docker build → tag with git SHA → docker push → deploy K8s with new image tag.
⚓
Kubernetes for Data Engineers
Pod
Smallest deployable unit. 1+ containers sharing network namespace and storage volumes. Pods are ephemeral — they can be killed and rescheduled at any time.
Deployment
Manages a set of identical pods (replicas). Handles rolling updates, rollbacks. Use for Airflow webserver, API services. kubectl rollout status deployment/airflow-web
ConfigMap / Secret
Externalize configuration and credentials. secretKeyRef injects secret as env var. Secrets can sync from AWS Secrets Manager via External Secrets Operator.
KubernetesExecutor
Airflow launches a K8s pod per task. Complete isolation, scales to zero, no worker fleet to manage. Each task gets its own CPU/memory. Production standard at scale.
Spark on K8s
Driver pod + dynamically provisioned executor pods. spark.kubernetes.container.image. Use Spark Operator (Google) for declarative Spark job management.
Helm
Package manager for K8s. Official Helm charts for Airflow, Kafka (Bitnami), Spark. Parameterize with values.yaml. Manage upgrades: helm upgrade --install.

Data Quality & Governance

Data that can't be trusted is worse than no data. Quality and governance are engineering, not administration.

01
Data Quality Framework
⚠️
The silent killer: Most data quality issues don't crash pipelines — they silently produce wrong results that analysts trust for months. Data quality monitoring must be automated and built into the pipeline, not checked manually after the fact.
✅
The 5 Dimensions of Data Quality
Freshness
Was the data updated on time? Check MAX(updated_at) > NOW() - INTERVAL '24 hours'. Set freshness SLAs per table. Alert when violated.
Completeness
Are expected columns populated? Null rates should be stable. Alert on sudden spikes in null rates — often indicates upstream schema change or pipeline failure.
Validity
Are values within expected ranges? Revenue > 0. Email addresses match regex. Dates are not in the future. Categorical columns match accepted set of values.
Consistency
Does data agree across systems? Total orders in orders table = sum of order lines. Revenue in sales DB = revenue in warehouse. Cross-table referential integrity.
Uniqueness
Are expected unique keys actually unique? Duplicate order IDs are catastrophic. Run COUNT(*) != COUNT(DISTINCT id) checks after every pipeline run.
🧪
Tooling
Great Expectations
Python library. Define Expectation Suites, run against DataFrame or SQL. Generates HTML Data Docs. Integrates as Airflow operator. Rich built-in expectations library.
Soda Core
YAML-based data checks. Simpler to write than GE, directly against SQL. soda scan in CI/CD. Soda Cloud for centralized monitoring and alerting dashboard.
dbt Tests
For warehouse tables: built-in + dbt-expectations. Run in CI. dbt build = models + tests together. First line of defense for transform layer correctness.
Monte Carlo
ML-based data observability SaaS. Automatically detects anomalies in volume, schema, and distributions without writing explicit rules. Generates lineage automatically.
Custom Monitors
Row count trends, null rate time-series, distribution drift detection (KL divergence / Z-score). Store daily metrics in monitoring table. Alert via PagerDuty/Slack webhook.
02
Data Contracts
📄
Data Contracts are formal agreements between data producers (engineers who generate data) and data consumers (analysts, other pipelines). They define: schema, SLA (freshness), quality expectations, and ownership. Violations break the pipeline early — before bad data reaches downstream consumers.
# Data Contract Example (YAML) name: orders version: "2.1.0" owner: data-platform@company.com schema: - name: order_id type: STRING nullable: false unique: true - name: customer_id type: STRING nullable: false - name: amount type: DECIMAL minimum: 0 - name: created_at type: TIMESTAMP nullable: false quality: - rule: row_count_minimum value: 1000 # per day - rule: null_rate_maximum column: customer_id value: 0.0 - rule: freshness_maximum value: "6 hours" sla: "08:00 UTC daily"
03
Apache Iceberg — Deep Dive
🧊
Iceberg Architecture
Catalog Layer
Stores pointer to current metadata. Options: Hive Metastore, AWS Glue Catalog, REST Catalog, JDBC. Enables atomic table swaps (pointer update is atomic).
Metadata Files
JSON files tracking table schema history, partition specs, snapshot history, sort orders. Each write creates a new metadata file. Enables schema/partition evolution without rewriting data.
Manifest Lists
Each snapshot has a manifest list file referencing multiple manifest files. Stores partition-level stats (min/max values) for partition pruning without reading data files.
Manifest Files
Track which data files belong to a snapshot. Store column-level stats per file (null count, lower/upper bounds) — enables aggressive file skipping at query time.
Data Files
Actual Parquet (default), Avro, or ORC files in your object store. Immutable once written. Old files cleaned up by expire_snapshots + remove_orphan_files procedures.
✨
Key Capabilities
Time Travel
SELECT * FROM orders FOR SYSTEM_TIME AS OF '2025-01-01' or VERSION AS OF 12345. Query any historical snapshot. Great for debugging bad data loads.
Schema Evolution
Add, rename, reorder, drop, promote-to-struct columns — without rewriting data. Column IDs (not names) track columns. Old files remain valid.
Partition Evolution
Change partition strategy (e.g. day → hour) without migrating existing data. Old data uses old partition spec; new data uses new spec. Transparent to query engines.
Row-Level Changes
MERGE INTO, UPDATE, DELETE — previously impossible on data lakes. Stored as: copy-on-write (rewrites files) or merge-on-read (append delete files, merge at read).
ACID Transactions
Optimistic concurrency control. Concurrent writers use snapshot isolation — each sees consistent view. Commit conflict detection prevents lost updates.

Tools & Technologies

Every major tool in the data engineering ecosystem — categorized and evaluated.

01
Ingestion & Integration
🔌
Fivetran
Managed ELT Connector
500+ pre-built connectors. Automated schema migration, incremental syncs, built-in transformations. Best option when budget allows — saves weeks of pipeline work.
✦ When: moving quickly, many SaaS sources
🌊
Airbyte
Open-Source ELT
OSS Fivetran alternative. Self-hosted or Airbyte Cloud. Python/low-code Connector Builder for custom sources. 300+ community connectors.
✦ When: cost-sensitive, custom sources needed
📡
Apache Kafka
Event Streaming Platform
The industry standard for real-time event streaming. Durable, scalable, fault-tolerant. Confluent Cloud for managed. Use Kafka Connect for source/sink connectors.
✦ When: real-time, event-driven, >100K events/sec
🔄
Debezium
Change Data Capture
Captures row-level database changes from MySQL, PostgreSQL, MongoDB, Oracle via binlog/WAL. Publishes to Kafka topics. Real-time CDC with exactly-once guarantees.
✦ When: real-time DB replication, CDC pipelines
🪣
AWS Glue
Serverless ETL (AWS)
Serverless Spark ETL + Glue Data Catalog (Hive-compatible metadata store). Crawlers auto-discover schemas from S3. DynamicFrames for flexible schema handling.
✦ When: AWS-native, serverless Spark needed
🌊
Dataflow (GCP)
Managed Apache Beam
Fully managed batch + streaming on Apache Beam. Unified programming model — same code for batch and stream. Auto-scaling, no cluster management.
✦ When: GCP ecosystem, unified batch+stream
🏭
Azure Data Factory
Azure ETL/Orchestration
Visual low-code ETL pipelines. 100+ built-in connectors. Mapping Data Flows (Spark-based). Integrates with Synapse, ADLS. Dominant in Microsoft/enterprise shops.
✦ When: Azure-native data platform
🔗
Singer/Meltano
Open-Source Tap/Target
Singer is a spec for source taps and destination targets. Meltano wraps Singer with project management, CI/CD. Free alternative to Fivetran for small teams.
✦ When: open-source, self-hosted, Singer ecosystem
02
Orchestration
🌀
Apache Airflow
Industry Standard Orchestrator
Python DAGs, huge provider ecosystem, mature UI, MWAA/Cloud Composer/Astronomer managed. Steep learning curve. Default choice for most companies.
✦ When: team knows Python, complex dependencies
🌊
Prefect
Modern Workflow Orchestration
Python-first, dynamic workflows, excellent DX vs Airflow. Decorators-based (@flow, @task). Prefect Cloud for UI + monitoring. No YAML needed.
✦ When: Airflow feels heavyweight, Python-centric team
⚙️
Dagster
Asset-Based Orchestration
Think in assets (tables, files, models) not tasks. Built-in lineage, partitioning, backfill UI, type system. Excellent for dbt + Spark integrations.
✦ When: asset-centric thinking, modern data stack
🔱
Luigi (legacy)
Task Pipeline Framework
Spotify's original pipeline framework. Dependency-based, Python tasks, simple file targets. Largely superseded by Airflow/Prefect but still in legacy codebases.
✦ When: maintaining legacy Luigi pipelines
03
Processing & Query Engines
⚡
Apache Spark
Distributed Batch + Streaming
The dominant distributed engine. PySpark API, Spark SQL, Structured Streaming. Databricks for managed. MLlib for ML at scale. Industry standard for large-scale ETL.
✦ When: >1TB data, Databricks environment
🌊
Apache Flink
True Stream Processing
Stateful stream processing with millisecond latency. Better streaming semantics than Spark. Flink SQL, complex event processing. Used by Uber, Alibaba, LinkedIn.
✦ When: low-latency stream processing, complex state
🦆
DuckDB
Embedded OLAP Engine
In-process columnar analytics. Query Parquet/CSV/JSON with SQL. No cluster, no setup. 10–100× faster than Pandas for analytical queries. The best kept secret in data engineering.
✦ When: <50GB, local dev, pipeline testing, EDA
🔍
Trino (Presto)
Federated SQL Query Engine
Query across Hive, S3 (Iceberg/Hudi/Delta), Cassandra, Kafka, RDBMS with a single SQL. MPP architecture, sub-minute interactive queries on data lake. Starburst for enterprise.
✦ When: query federation, interactive lake queries
🐻
Polars
Fast DataFrame Library
Rust-based DataFrame library for Python. Lazy evaluation with query optimizer, multi-threaded. 5–50× faster than Pandas. Arrow-native. Ideal for single-node pipeline processing.
✦ When: replace Pandas, single-node but large data
🗄️
ClickHouse
Real-Time Analytics DB
Columnar OLAP DB for real-time analytics. Blazing fast aggregations (billions rows in milliseconds). Ideal for observability, time-series, user-facing analytics. ClickHouse Cloud managed option.
✦ When: real-time analytics serving, sub-second queries
04
Warehouses & Lakehouse
❄️
Snowflake
Cloud Data Warehouse
Separate compute/storage, instant scale, zero-copy cloning, data sharing, multi-cloud (AWS/GCP/Azure). Snowpark for Python/Java UDFs. Market leader in cloud DW.
🔵
BigQuery
Serverless Warehouse (GCP)
Serverless — no cluster to manage. Per-query ($5/TB) or flat-rate pricing. Excellent at scale, Omni for multi-cloud, BQML, BigQuery Omni, native Iceberg support.
🔶
Databricks
Lakehouse Platform
Spark + Delta Lake + MLflow + Unity Catalog. Delta Sharing for data exchange. Photon engine (C++ vectorized execution). Market leader in unified analytics.
🟠
Amazon Redshift
Cloud MPP Warehouse
Columnar MPP warehouse. RA3 nodes with managed storage. Redshift Spectrum for S3 queries. Serverless Redshift option. Deep AWS ecosystem integration.

Interview Preparation
📝
SQL Interviews — What to Expect
Window Functions
RANK, DENSE_RANK, ROW_NUMBER, LAG/LEAD, running totals, moving averages. "Find users who logged in 3+ consecutive days" = classic window function problem.
Complex JOINs
Self joins, non-equi joins, LEFT JOIN with NULLs, JOIN vs subquery performance. "Find customers with no orders" — know multiple approaches.
Aggregations
GROUP BY + HAVING, ROLLUP/CUBE, GROUPING SETS. Percentile functions (PERCENTILE_CONT). Median calculation without built-in function.
Practice Resources
DataLemur (DE-specific), StrataScratch (real interview questions), LeetCode SQL (Hard), Mode Analytics SQL Tutorial, pgexercises.com
🏗️
System Design — Common Questions
Real-time pipeline
"Design a pipeline to ingest 1M events/sec from mobile apps and make them queryable in <60 seconds." Answer: Kafka → Flink/Spark Streaming → ClickHouse/BigQuery.
Batch warehouse
"Design a daily ETL pipeline for 50 source systems into a data warehouse with SLA of 8AM delivery." Cover: ingestion, orchestration, monitoring, failure recovery.
Data modeling
"Model an e-commerce data warehouse. What fact and dimension tables would you create?" Walk through: define grain, identify dims (customer, product, date, geography), define SCD types.
Trade-off thinking
Interviewers want you to articulate: batch vs streaming trade-offs, normalized vs denormalized, cost vs performance, build vs buy. There's no single right answer — show reasoning.

Portfolio Projects to Build
  • ◆
    End-to-End Batch Pipeline: Ingest a public dataset (NYC Taxi, Spotify API, GitHub Archive) → S3/GCS → PySpark/dbt transform → Snowflake/BigQuery → Metabase dashboard. Deploy Airflow in Docker. Add Great Expectations tests.
  • ◆
    Real-Time Streaming Pipeline: Produce events to Kafka → Flink/Spark Structured Streaming → aggregate into ClickHouse or BigQuery → live Grafana/Metabase dashboard. Use crypto prices, weather API, or fake event generator.
  • ◆
    Lakehouse on S3: Create Apache Iceberg or Delta tables on S3. Demonstrate time-travel queries, schema evolution (add/rename columns), MERGE operations, partition evolution. Document performance vs plain Parquet.
  • ◆
    dbt + Airflow Project: Full ELT stack — Airflow triggers Fivetran sync → dbt run/test → Great Expectations validation → Slack alert on failure. Apply Kimball dimensional modeling. Deployed on Docker Compose.
  • ◆
    Infrastructure as Code: Terraform a complete cloud data stack (S3 + Glue + Redshift or GCS + BigQuery + Dataproc). GitHub Actions pipeline: lint → test → apply. Demonstrate state management and workspace separation (dev/prod).
  • ◆
    Data Quality Framework: Build comprehensive monitoring on an existing pipeline — row count trends, null rate monitoring, schema change detection, distribution drift alerts. Custom Grafana dashboard. Publish a blog post explaining your approach.

Career Progression
📈
Individual Contributor Track
Junior (0–2y)
Execute well-scoped tasks. Build pipelines from templates. Fix bugs. Learn the existing stack. Contribute to code reviews. Ask many questions — this is expected.
Mid-Level (2–5y)
Own feature areas end-to-end. Design pipeline architecture for new data sources. Mentor juniors. Handle on-call incidents. Drive technical decisions for your domain.
Senior (5–8y)
Define technical strategy for multiple areas. Drive cross-team technical initiatives. Identify systemic problems before they occur. Write design docs. Interview and hire candidates.
Staff (8–12y)
Org-wide technical decisions. Evaluate and introduce new technologies. Lead multi-quarter migrations. Set platform standards. Influence company data strategy.
Principal (12y+)
Industry-level impact. Open source leadership. Architectural vision for entire data org. External conference speaking. Shape the direction of data engineering practices.
🌿
Adjacent Career Paths
Analytics Engineer
dbt-focused role bridging DE and analytics. Owns semantic layer, data models, metrics definitions. Less infrastructure, more business logic. Growing rapidly in demand.
ML Engineer
Feature pipelines + model training infra + model serving. Combines DE skills with ML knowledge. Builds MLOps platforms (feature stores, experiment tracking, model registries).
Data Platform Eng.
Build and operate internal data infrastructure — managed Airflow, dbt platform, data catalog, cost management tooling. Less pipeline building, more platform for other DEs.
Data Architect
Enterprise-level design, vendor evaluation, governance strategy, migration planning, standards definition. Often a senior IC or principal who specializes in architecture over implementation.
Engineering Manager
Lead a team of 4–8 DEs. Technical credibility + people skills. Own team roadmap, hiring, performance reviews. Stop coding day-to-day, become a force multiplier.
